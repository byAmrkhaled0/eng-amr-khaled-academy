# تقرير فحص وتسليم Techno Minds V60.6.0

تاريخ الفحص: 16 يوليو 2026  
المصدر المفحوص: Techno Minds V60.5.0  
النتيجة: نسخة V60.6.0 مبنية ومتحقق منها محليًا، ولم يُنفذ أي نشر أو حذف أو Migration على Firebase أو GitHub.

## المشاكل الفعلية التي وُجدت وتم إصلاحها

| المجال | المشكلة في V60.5.0 | المعالجة في V60.6.0 |
|---|---|---|
| المدفوعات | `paid/paymentAmount` يمثلان لقطة واحدة ويمكن أن يستبدلا تاريخ الشهر السابق | دفتر `monthly_payments` وعمليات مستقلة غير محذوفة داخل `payment_transactions` |
| الدفع الجزئي | لا يوجد تجميع موثوق لعدة دفعات لنفس الشهر | جمع ذري داخل Firestore Transaction وحساب المطلوب والمدفوع والمتبقي والحالة |
| التكرار | الضغط أو إعادة الطلب قد يسجلان العملية مرتين | `requestId` ثابت للمحاولة وسجل `_payment_dedup` مؤقت على الخادم |
| التعديل والإلغاء | لا يوجد أثر تدقيق كامل | Admin فقط؛ الإلغاء يغيّر الحالة ولا يحذف العملية، وكل إجراء يسجل في `activityLog` |
| البيانات القديمة | لا توجد Migration آمنة | Backup سحابي قبل الكتابة، معرّفات حتمية، تخطي السجل الموجود/الصفري، وعدم حذف الحقول القديمة |
| الخزنة | الإجماليات مبنية على الحالة القديمة ولا تغطي الجزئي | مقبوضات اليوم والفترة والمتوقع والمتبقي وأعداد كامل/جزئي/غير مدفوع وإجماليات لكل مسار |
| التحديث اللحظي | صفحة الدفع تعتمد على إعادة رسم/حفظ قديم | Listeners مخصصة للمدفوعات فقط، وتُلغى عند مغادرة القسم أو تسجيل الخروج |
| سجل الطالب | لا يعرض تاريخ الشهور | تبويب مدفوعات في بوابة الطالب وولي الأمر وداخل ملف الإدارة |
| Push | تنبيه أثناء فتح اللوحة فقط | FCM Token لكل جهاز إدارة، Trigger عند الحجز، Push خلفي، تعطيل التوكن المنتهي وإلغاؤه عند الخروج |
| Service Worker | مسار Push غير مكتمل وخطر تكرار مشاكل CDN/importScripts | Push API مباشر داخل العامل نفسه بلا `importScripts`، وكاش إصدار V60.6 مستقل |
| نشر Windows | رسالة نجاح قد تظهر بعد توقف، ولا يوجد استكمال أو تحديد Function | Success marker، كود خروج صحيح، `-Resume`، نشر Function منفردة، و`FUNCTIONS_DISCOVERY_TIMEOUT=120` |
| GitHub | مسار التجهيز كان معرضًا لمشكلة Execution Policy وخلط النشر | Wrapper بـBypass، Unblock، نسخ محلي آمن، واستبعاد `node_modules` و`dist` وكل `.env*` السرية؛ لا Push تلقائي |
| الأداء | JPEG رئيسية أكبر، وضرورة التأكد من عدم تحميل QR/Excel مقدمًا | WebP بحجم يقارب 102KB مع JPEG fallback؛ QR وExcel تحميل كسول ولا يدخلان App Shell |
| الهاتف | صفحة الخزنة لم تكن مناسبة كسجل معاملات كثيف | بطاقات Mobile-first وفلاتر شبكية وحوار دفع أحادي العمود تحت 600px ومنع التمدد الأفقي |
| CSV/Excel | البيانات النصية قد تبدأ بعلامة Formula | تحييد القيم التي تبدأ بـ`= + - @` قبل التصدير |
| طبقات JavaScript | V53/V55 تعيدان تعريف واجهة الدفع بعد `DOMContentLoaded` | مسجل V60.6 يثبت معالجاته بعد hooks التوافق، ويحوّل أي استدعاء `setPaid` قديم للنموذج الجديد بدل كتابة المرآة |

## تغييرات Firebase والأمان

- القراءة المباشرة لبيانات الطلاب والمحتوى المقيد ما زالت لفريق العمل فقط؛ بوابات الطالب تستخدم Callables تتحقق من الكود والمسار وتطبق Rate Limiting.
- الكتابة على `monthly_payments` و`payment_transactions` و`_payment_dedup` مغلقة من الواجهة وتتم عبر Cloud Functions فقط.
- إنشاء دفعة متاح للحسابات النشطة `admin/teacher/assistant`، بينما التعديل والإلغاء والترحيل للـ`admin` فقط.
- قواعد Storage تقيد أنواع وأحجام الملفات، ورفع واجب الطالب يحتاج تصريحًا مؤقتًا من الخادم.
- Judge0 Token غير موجود في الواجهة أو المصدر؛ مكانه `functions/.env` المستبعد من Git والحزمة.
- Firebase Web API key الموجود في `firebase-config.js` معرّف Web عام وليس Server secret؛ الحماية تعتمد على Auth وRules وFunctions.
- تمت إضافة فهارس المدفوعات وTTL لسجل منع التكرار، من دون حذف أي Index قديم من ملف النشر تلقائيًا.
- النسخ الاحتياطي والحذف الآمن يشملان مجموعتي المدفوعات الجديدتين.
- لم يوجد أي اسم أو شعار أو محتوى لمنصة محمود فوزي في المصدر المفحوص.

## اختبارات تتطلب إعدادًا خارجيًا ولم يُدّعَ نجاحها

- لم يتم إنشاء حجز حقيقي أو قبول/رفض حجز أو تسجيل/ترحيل دفعة داخل مشروع Firebase الحي؛ تنفيذ ذلك سيعدل بيانات حقيقية ويحتاج حسابًا مصرحًا.
- لم تُنشر Rules أو Indexes أو Functions أو Hosting، ولم يُشغل Migration، التزامًا بعدم تعديل Firebase قبل موافقة المالك.
- Push الخلفي يحتاج إنشاء Web Push certificate ووضع **المفتاح العام** في `messagingVapidKey` ثم اختبار جهاز حقيقي.
- تشغيل JavaScript وPython وC++ وJava وC# عبر Judge0 يحتاج Endpoint/حصة صالحة. السكربت `CHECK-SITE.cmd -FullCodeRunner` ينفذ الاختبار الحقيقي بعد النشر.
- سكربتات PowerShell فُحصت بنيويًا ونصيًا، لكن لم تُنفذ في بيئة Windows/PowerShell 5.1 لأن بيئة الفحص Linux لا تحتوي PowerShell.
- لم يُشغل Lighthouse أو اختبار متصفح بصري آلي لعدم توفر Chromium. تم فحص breakpoints ‏320/360/375/390/430 ساكنًا، ومراجع HTML وCSS والبناء، لكن يلزم تشغيل Lighthouse واختبار لمس فعلي بعد النشر.
- لم يمكن اختبار Authentication بالأدوار أو رفع ملف حي أو CORS/403/500 الحقيقي من دون جلسات وحسابات وبيئة Firebase المنشورة.

## ترتيب التحقق بعد موافقة المالك على النشر

1. إضافة VAPID public key وإعداد Judge0 في `functions/.env` المحلي.
2. تشغيل `npm test` و`npm run build` و`npm run verify:dist` على Node.js 22.
3. تشغيل `DEPLOY-WINDOWS.cmd`؛ وعند تعطل خطوة استخدام `DEPLOY-WINDOWS.cmd -Resume`.
4. تشغيل `CHECK-SITE.cmd` ثم `CHECK-SITE.cmd -FullCodeRunner`.
5. بحساب Admin: إنشاء حجز اختباري وقبوله ورفض آخر، والتحقق من المجموعة والبوابات والمسار.
6. إنشاء دفعة جزئية ثم كاملة، تعديل عملية وإلغاء أخرى، ومطابقة الخزنة والتصدير.
7. أخذ Backup ومراجعة أسعار/شهور البيانات القديمة قبل تشغيل Migration من زر المدفوعات.
8. بعد نجاح المراجعة فقط، تجهيز GitHub بواسطة `PREPARE-GITHUB.cmd` ومراجعة `git status` قبل Commit/Push يدوي.

هذا التقرير يثبت الاختبارات المحلية المذكورة فقط، ولا يصف الخدمات الخارجية بأنها ناجحة قبل تشغيل فحص ما بعد النشر.
